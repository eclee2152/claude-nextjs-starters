# AI 기반 출입관리 시스템 MVP PRD

**버전**: 2.0.0
**작성일**: 2026-04-10
**변경 이력**: v1.0.0 → v2.0.0 (멀티테넌시 전략 변경: RLS 컬럼 분리 → 스키마 복제 방식)

---

## 핵심 정보

**목적**: 기업별 독립 스키마 기반으로 방문자 사전 교육, QR 출입 인증, 실시간 출입 이력을 통합 관리하는 SaaS 출입관리 플랫폼

**사용자**: 보안 관리가 필요한 기업의 보안담당자, 임직원, 외부 방문자

---

## 핵심 변경사항 (v1.0.0 → v2.0.0)

| 항목 | v1.0.0 | v2.0.0 |
|------|--------|--------|
| 멀티테넌시 전략 | 모든 테이블에 company_id 컬럼 + Supabase RLS | 회사별 독립 PostgreSQL 스키마 복제 |
| 데이터 격리 방식 | RLS 정책으로 행(Row) 단위 격리 | 스키마 단위 물리적 격리 |
| 스키마 구조 | public 스키마 단일 사용 | public (공통) + {company_slug} (회사별) 분리 |
| company_id 컬럼 | 모든 테이블에 존재 | 회사별 스키마 내 테이블에는 불필요 |

## 사용자 역할 (v1.0.0 유지)

| 역할 | 설명 | 접근 범위 |
|------|------|----------|
| 슈퍼 관리자 | 플랫폼 전체 관리자 | 전체 회사 관리, 스키마 생성/삭제 |
| 회사 관리자 | 회사 등록 시 자동 지정되는 메인 관리자 | 자사 사용자 관리, 방문 승인 |
| 임직원 | 회사 관리자가 등록한 직원 | 방문 신청 승인/반려 |
| 보안담당자 | 회사 관리자가 등록한 보안 인력 | QR 스캔, 출입 기록 관리 |
| 방문자 | 사전 등록 없이 방문 신청하는 외부인 | 방문 신청, 교육 이수, QR 수령 |

## 사용자 여정

```
1. [슈퍼 관리자] 회사 관리 페이지
   ↓ 회사 등록 폼 제출

2. [시스템 자동 처리]
   - PostgreSQL에 {company_slug} 스키마 생성 (base 스키마 복제)
   - public.companies에 회사 row 추가
   - {company_slug}.users에 회사 관리자 계정 자동 생성
   - 도메인 할당 ({company_slug}.service.com)
   ↓

3. [회사 관리자] 사용자 관리 페이지 (로그인 후)
   ↓ 임직원/보안담당자 등록

4. [방문자] 방문 신청 페이지
   - 방문 정보 입력 (이름, 성별, 생년월일, 연락처, 업체명, 차량번호, 물품, 담당 임직원 선택)
   ↓ 신청 완료

5. [방문자] 교육 이수 페이지
   - 교육 영상 시청
   ↓ 영상 시청 완료

   - 문제풀이
   ↓ 합격 시 → QR 생성 페이지
     불합격 시 → 재시도 (문제풀이 페이지 유지)

6. [방문자] QR 코드 페이지
   - QR 코드 발급 (승인 대기 상태)
   ↓

7. [임직원] 방문 승인 페이지
   - 방문 신청 목록 조회
   - 리스트 더블클릭 → 승인/반려 팝업
   ↓ 승인 → 방문자 QR 활성화
     반려 → 방문자 출입 불가 처리

8. [보안담당자] 출입 조회 페이지
   - QR 리더기로 QR 스캔
   ↓ 승인된 방문자 → 入 기록 생성 + 출입 허용
     미승인/반려 방문자 → 출입 불가 표시

9. [보안담당자] 출입 조회 페이지
   - 출입 목록의 출문 버튼 클릭 → 出 기록 생성
   ↓

10. [완료] → 출입 이력 확인 가능
```

## 기능 명세

### 1. MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|-------------|------------|
| **F001** | 회사 등록 및 스키마 생성 | 회사 정보 입력 후 독립 PostgreSQL 스키마 자동 생성 | SaaS 멀티테넌시 핵심 | 회사 관리 페이지 |
| **F002** | 사용자 등록 | 임직원/보안담당자 계정 생성 (역할 구분) | 방문 승인 및 출입 관리 인력 구성 | 사용자 관리 페이지 |
| **F003** | 방문 신청 | 방문자 정보 입력 및 담당 임직원 선택 | 출입관리의 시작점 | 방문 신청 페이지 |
| **F004** | 교육 영상 시청 | 방문 전 의무 교육 영상 재생 및 완료 확인 | 보안 교육 의무화 | 교육 이수 페이지 |
| **F005** | 문제풀이 및 합격 처리 | 교육 후 문제풀이, 합격 시 QR 생성 가능 상태로 전환 | 교육 이수 확인 | 교육 이수 페이지 |
| **F006** | QR 코드 생성 및 발급 | 교육 이수 완료 후 방문자 식별 QR 코드 생성 | 출입 인증 수단 | QR 코드 페이지 |
| **F007** | 방문 승인 및 반려 | 임직원이 방문 신청 목록 조회 후 승인/반려 처리 | 방문 허가 결정 | 방문 승인 페이지 |
| **F008** | QR 스캔 및 출입 기록 | 보안담당자가 QR 읽기 후 入/出 기록 생성 | 실시간 출입 이력 관리 | 출입 조회 페이지 |

| **다음 이동** | 출입 기록 실시간 반영 후 목록 갱신 |

## 데이터 모델

### 멀티테넌시 스키마 구조

```
PostgreSQL (Supabase)
├── public 스키마 (공통 - 전체 테넌트 공유)
│   └── companies          ← 회사 목록 및 스키마 메타데이터
│
├── base 스키마 (복제 기준 템플릿 - 직접 사용 안 함)
│   ├── users
│   ├── visit_requests
│   ├── approvals
│   └── access_logs
│
├── {company_slug_A} 스키마 (A사 전용)
│   ├── users
│   ├── visit_requests
│   ├── approvals
│   └── access_logs
│
└── {company_slug_B} 스키마 (B사 전용)
    ├── users
    ├── visit_requests
    ├── approvals
    └── access_logs
```

### Supabase 구현 방식

회사 등록 시 서버에서 다음 SQL을 실행합니다.

```sql
-- 1. 신규 스키마 생성
CREATE SCHEMA IF NOT EXISTS {company_slug};

-- 2. base 스키마의 테이블 구조 복제
CREATE TABLE {company_slug}.users (LIKE base.users INCLUDING ALL);
CREATE TABLE {company_slug}.visit_requests (LIKE base.visit_requests INCLUDING ALL);
CREATE TABLE {company_slug}.approvals (LIKE base.approvals INCLUDING ALL);
CREATE TABLE {company_slug}.access_logs (LIKE base.access_logs INCLUDING ALL);

-- 3. 해당 회사 서비스 요청 시 search_path 동적 설정
SET search_path TO {company_slug}, public;
```

미들웨어(Next.js Middleware)에서 요청 도메인을 파싱하여 company_slug를 추출하고, Supabase 쿼리 실행 전 search_path를 동적으로 설정합니다.
